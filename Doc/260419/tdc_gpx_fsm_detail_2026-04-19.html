1. 설계 개요와 시간 축

TDC-GPX Controller 는 ams TDC-GPX 칩을 제어하여 Laser LiDAR 측정을 수행하고, 결과를 VDMA 를 통해 DDR 프레임으로 전달한다. 동작은 다음 단계로 구분된다:

Phase단계주요 모듈 (leaf)지속 시간 (@200MHz 기준)
0HW reset → bus 프로토콜 준비bus_phy (대기 상태)수 µs
1SW 초기 CSR 설정csr_pipeline, csr_chip수 µs (PS 액세스)
2Chip powerup / cfg_write (per chip)chip_init → bus_phy~240 ns (PuResN) + config writes
3cmd_start → face_seq 승인face_seq, cmd_arb, chip_ctrl수 cycle
4Shot capture → drain → ALUchip_run, stop_cfg_decoderound-trip + drain (~µs)
5Raw → event decodedecoder_i_mode, raw_event_builder2 cycle/beat
6Cell 조립 (per chip)cell_builderbeats/cell × 1 clk
7Row 조립 (4 chip in-order)face_assemblerrow 당 1 + Σcell beats
8Header prefix + VDMA 출력header_inserter6 또는 12 beat 헤더 + row

3.2 tdc_gpx_csr_pipeline / csr_chip leaf s_axi_aclk → i_axis_aclk

FSM

3.5 tdc_gpx_cmd_arb leaf i_axis_aclk

multi-chip register access 중재. cfg_write gating + per-chip busy 추적 + 1-depth overlap queue.
FSM: 명시적 없음 (combinational + pending latch)

로직

cmd_arb 는 상태 머신 대신 per-chip 병렬 dispatch 구조:

  • SW 의 reg_read/write 요청을 chip_address (2-bit) + chip_busy 조건으로 chip 0-3 중 하나에 라우팅
  • chip 이 busy 면 pending latch (1-depth) 에 보관
  • dispatch-wait watchdog (16-bit) → 타임아웃 시 reject
  • cfg_write gating: face_asm_idle=1 + reg_outstanding=0 일 때만 허용
  • per-chip reg_outstanding tracker (4-bit)

Error sticky

  • o_reg_timeout, o_reg_rejected, o_reg_zero_mask, o_err_rw_ambiguous

Raw FIFO (Round 13 P1)

  • Depth 6, t_raw_entry: {valid, tdata[31:0], tuser[7:0], drain}
  • Data beat: 4 slot 이상 free 일 때만 enqueue
  • Control beat (tuser[7]=1): 1 slot 있으면 enqueue
  • FIFO full + control beat → 가장 오래된 data beat 를 evict
  • tuser[5] = faulted bit (final drain_done only), 하위로 전달

  • o_drain_done, o_run_drain_complete, o_drain_done_faulted
  • o_err_{raw_drop, drain_cap, bus_fatal, force_reinit, ...} (sticky)

3.10 tdc_gpx_decoder_i_mode leaf i_axis_aclk

28-bit raw IFIFO word → hit 필드 추출 + control beat pass-through.
FSM: 없음 (single-stage registered)Latency: 1Throughput: 1 beat/clk

처리 (combinational + 1 register stage)

if tuser[7] = 1:  -- control beat
    passthrough: drain_done flag, ififo_id → tuser[6], faulted → tuser[5]
else:  -- data beat
    tdata[16:0]    <- raw[16:0]   (hit 17-bit)
    tuser[0]       <- raw[17]     (slope)
    tuser[2:1]     <- raw[27:26]  (cha_code)
    tuser[5:3]     <- ififo_id & raw[27:26]  (stop_id_local)
    tuser[6]       <- ififo_id
    tuser[7]       <- 0

I/O

  • in: i_s_axis_{tvalid, tdata[31:0], tuser[7:0]}; o_s_axis_tready
  • out: o_m_axis_{tvalid, tdata[31:0], tuser[7:0]}; i_m_axis_tready
  • storage: s_tvalid_r, s_tdata_r[31:0], s_tuser_r[7:0] (1-stage)

3.11 tdc_gpx_raw_event_builder leaf i_axis_aclk

chip_id 태그 + per-stop hit_seq 누적 카운터 + out-of-range stop_id 검출.
FSM: 없음Latency: 1

처리

if tuser[7] = 1 and tuser[6] = 1:  -- final drain_done → reset all counters
    s_hit_cnt_r <= (others => 0)
elseif data beat and stop_id < stops_per_chip:
    increment s_hit_cnt_r(stop_id)
    output event with hit_seq, chip_id, shot_seq tags

output tuser[15:0]:
    [0]    = slope
    [2:1]  = chip_id
    [5:3]  = stop_id_local
    [5]    = faulted (control beat only)
    [6]    = ififo_id
    [7]    = drain_done
    [10:8] = hit_seq_local (3-bit per stop)
    [15:11] = shot_seq[4:0]

메모리

  • s_hit_cnt_r: 8 entries × 3-bit = 24 FF per chip
  • 1-stage output register (16-bit tuser, 32-bit tdata)

시간 순 배선 진행

  1. HW reset → 모든 서브 reset
  2. csr_pipeline + csr_chip 이 AXI4-Lite 로 config 수집, CDC 로 i_axis_aclk 로 전달
  3. config_ctrl 이 chip_init 시작 (PuResN + REG writes)
  4. face_seq 가 cmd_start 수락 → shot_start_gated pulse
  5. config_ctrl 의 chip_run 이 shot 수행 → raw_cdc 로 Cluster 2 전달
  6. decode_pipe → cell_pipe → output_stage 파이프라인 흐름
  7. header_inserter 가 VDMA frame 출력
  8. status_agg 가 end-to-end sticky/counter 를 status record 로 조립
  9. err_handler / face_seq 가 사이드 경로로 에러 복구 / 프레임 경계 관리
